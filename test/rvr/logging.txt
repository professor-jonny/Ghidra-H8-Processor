CONFIRMED - MECHANISM (how the MUT table resolves an address)
=========================================================================

[VERIFIED] MUT table lookup mechanism, decompiled directly:
  Function: adc_sensor_convert_single @ ROM 0x171c3
  For a given RequestID (index):
    1. Compute ROM pointer-table address = 0x2fad0 + (index * 2)
    2. Read a 16-bit value from that ROM address -> this is a RAM address
    3. Floor-clamp the RAM address to 0xEE80
    4. Read and return the byte at that RAM address
  This is ONE level of indirection: the MUT table stores pointers to RAM
  cells, not the values themselves. This matches what the logging profile
  assumed.

[VERIFIED] Raw ROM bytes checked directly against two claimed addresses:
  - RequestID 0x21 (claimed RAM 0xF17B): ROM bytes at 0x2fad0+0x42 = F1 7B. MATCH.
  - RequestID 0x17 (claimed RAM 0xF13D): ROM bytes at 0x2fad0+0x2E = F1 3D. MATCH.
  => The address ARITHMETIC used to build this logging profile is correct.
     This does NOT confirm the RequestID's semantic meaning (see below).

CONFIRMED - REAL SENSOR READ FUNCTION FOUND

[VERIFIED] adc_read_sequence_main @ ROM 0x1556d is the actual round-robin
analog sensor reader, called from sensor_adc_update @ 0x211fb, called from

[CRITICAL - UNRESOLVED CONFLICT] 0xF3AC/0xF3AE: TWO subsystems, same RAM
=========================================================================

[CONFIRMED - GENUINE CONFLICT, NOT A STALE LABEL] Checked xrefs to 0xF3AC
directly (not just knock_process_main's usage):

  From 00024f17 in tcu_slip_control_init [WRITE]  (init only)
  From 00029afe/00029b21/00029b7d/00029b8a/00029bd9/00029be6 in
       knock_process_main [READ/WRITE, heavy, ongoing]
  From 0002b5bd in tcu_rx_main_scheduler [READ]
  (tcu_rx_main_scheduler ALSO writes 0xF3AC/0xF3AE directly at 0xb5cf/
   0xb5ec-ish, using tcu_slip_ratio_raw * something / div_s32_s16_rounded
   - genuine TCU slip-ratio LOW-PASS FILTER math, ghidra global names
   already present: tcu_slip_ratio_raw, tcu_slip_ratio_filtered)

  knock_process_main's use of the SAME addresses is unambiguously
  different math: gain selected from 0xB08/0xB0A (per f3a4 bit7), input
  is an ADC-channel-6-derived per-cylinder value, output clamped 0-255
  via clamp_u16 and written back to 0xF3AC/0xF3AE inside an "if bit1 of
  0xf0ec (cylinder bank select) ..." branch structure - classic
  per-cylinder knock retard, not a slip ratio filter.

  CONCLUSION: 0xF3AC and 0xF3AE are read AND written by both
  tcu_rx_main_scheduler (TCU slip-ratio filter) and knock_process_main
  (per-cylinder knock retard) with mutually incompatible interpretations
  of the same bytes. This is either:
    (a) genuine RAM economy / cell reuse across subsystems that don't
        run concurrently in a way that matters (possible on this
        memory-constrained H8), or
    (b) one of the two functions is using a MISIDENTIFIED address that
        merely happens to be adjacent/aliased to the real one (e.g. off
        by a small constant not yet spotted), or
    (c) genuine shared coupling - e.g. knock retard on a cylinder
        legitimately feeds into TCU torque-cut coordination, and 0xF3AC/
        0xF3AE really is a shared communication cell by design.
  NOT RESOLVED. This directly affects confidence in RVR_base.xml's
  2025 reclassification of "TCU Torque Ramp Target A-D [f3a8/f3aa]" and
  the fuel-map document's "Open Loop Lambda Correction Bk1 [f3ac/f3ae]"
  labels - at minimum, f3ac/f3ae are NOT a clean single-purpose TCU
  register as currently documented; treat all f3a8/f3aa/f3ac/f3ae/f3b0/
  f3b2 table definitions in both project XMLs as suspect until this is
  resolved.

[RESOLVED 2026-07-02, SESSION 3] 0xF3AC/0xF3AE CONFLICT - CONFIRMED INTENTIONAL
=========================================================================

[RESOLVED] The f3ac/f3ae "conflict" between tcu_rx_main_scheduler (TCU slip-ratio
filter) and knock_process_main (per-cylinder knock retard) is NOT a bug, NOT a
misidentified address, and NOT incidental RAM reuse. It is CONFIRMED INTENTIONAL
cross-subsystem coordination. Resolution method: traced the enable conditions
guarding each subsystem's access to f3ac/f3ae.

  - knock_process_main's entire per-cylinder knock-retard block (which reads/
    writes f3ac/f3ae/f3aa via bank-selected calls to sat_mul_u16 @0x1405c and
    div_u16_rounded @0x1412b, gain from 0xB08/0xB0A, cylinder select via
    0xF0EC bit1) is gated by:
        if ((f1f2 & 0x80) == 0) return;

  - tcu_rx_main_scheduler's independent TCU slip-ratio low-pass filter (also
    writing f3ac/f3ae/f3aa via mul_u16_hi -> div_s32_s16_rounded) is gated by
    the IDENTICAL condition:
        if ((f1f2 & 0x80) != 0) { ...filter... }

  Both subsystems run under the SAME enable bit, not mutually exclusive states.
  This rules out possibility (a) from the original conflict note (safe reuse
  across non-concurrent states) and possibility (b) (misidentified/aliased
  address - the bit-level gating match is too precise to be coincidental).
  This CONFIRMS possibility (c): genuine shared coupling by design - knock
  retard on a cylinder legitimately feeds into TCU torque-cut coordination via
  this shared register, active whenever the current gear's config table
  enables it (see below).

  NEW FUNCTION IDENTIFIED: gear_state_config_loader_f1fc (renamed from
  FUN_00020f28). Derives a state index f1fc (0-7) from f15a (bucketed) or via
  ROM lookup table @0x222 indexed by f1fa. f1fc is the SAME gear/state
  selector already seen elsewhere in the firmware (the 4-byte-per-gear struct
  @ROM 0x232 used in sci1_meta_cmd_dispatch_c0_ff's 0xEC-0xEF branch, and the
  4-entry TCU Gear Ratio Periods table @ROM 0x3284) - consistent with f1fc
  being a genuine current-gear index (0-3, with 4-7 likely unused/reserved).

  f1fc is used to load THREE parallel per-gear config words from ROM:
    f1f0 <- ROM 0x262 + f1fc*2   (gear_config_table_f1f0, not yet decoded)
    f1f2 <- ROM 0x272 + f1fc*2   (gear_config_table_f1f2_TCU_KNOCK_COORD_ENABLE
                                   - bit7 is the shared knock/TCU coordination
                                   enable flag)
    f1f4 <- ROM 0x282 + f1fc*2   (gear_config_table_f1f4, not yet decoded)

  PRACTICAL INTERPRETATION: the ECU appears to enable knock-informed torque-cut
  coordination with the TCU only in certain gears (likely lockup/higher gears,
  pending confirmation of which gear indices have bit7 set in ROM table
  0x272). This is physically sensible: knock detected on a cylinder both
  retards timing AND requests a torque reduction via the TCU, protecting the
  drivetrain, but only where the transmission logic calls for it.

[SESSION 4, 2026-07-02] KNOCK RETARD DOES NOT EXIST AS A DISCRETE VALUE -
INVESTIGATING OCTANE LEVEL (MUT 0x27 / RAM 0xEEC9) AS THE REAL MECHANISM
=========================================================================

CONTEXT: Neither this ROM's MUT profile nor the stock Evo 4/5/6 profile
(Mitsubishi MUTII EFI.xml) has any RequestID for a discrete "knock retard
degrees" value. Working theory: on these ECUs knock isn't logged as a
subtracted degree value at all - it degrades "Octane Level" (starts at
100%, MUT RequestID 0x27, RAM word 0xEEC8/EEC9, low byte 0xEEC9 is what
MUT actually returns), and Octane Level is what blends between the
High-Octane and Low-Octane ignition maps already in RVR_base.xml
(High/Low Octane Ignition Map 1 & 2, Knock Control Above Load, Octane
Control Above Load). If true, "retard" is implicit in the map blend, not
a standalone logged scalar - explaining why no such MUT item exists.

[CROSS-VALIDATED] Knock Sum (RequestID 0x26 -> RAM 0xF3A3) and Knock
Voltage (RequestID 0x30 -> RAM 0xF15D) use IDENTICAL RequestIDs and Eval
formulas in the stock Evo 4/5/6 profile (Mitsubishi MUTII EFI.xml) as in
this RVR profile. Combined with the existing decompiled-write-site
evidence (knock_process_main writes both addresses directly), these two
items now have TWO independent confirming lines of evidence (address
arithmetic + write-site AND cross-model naming convention match). Upgrade
from CONFIRMED to CONFIRMED/CROSS-VALIDATED.

  (see session 3's "Knock Retard Per-Cyl" table cluster notes). This
  reinforces that this call chain is per-cylinder knock-condition
  bookkeeping, but neither function found so far actually writes Octane

[RESOLVED 2026-07-02, SAME SESSION] Full chain traced and CONFIRMED. The
EvoScan profile's claim ("Octane Level starts at 100% and decreases with
persistent knock sum") is CORRECT IN SUBSTANCE, via an intermediate
accumulator rather than a direct knock->octane write:

  1. knock_process_main (0x2933b) computes knock sum, writes RAM 0xF3A2
     (word) / 0xF3A3 (low byte = MUT Knock Sum, RequestID 0x26).
  2. knock_event_counter_update_eec6 (was FUN_000251c4, ROM 0x251c4),
     called from octane_level_gate_and_reset_dispatch's enabled-branch
     chain: reads 0xF3A2, compares against ROM-configured hysteresis
     thresholds (0xB3E low, 0xB40 high) gated by a cooldown timer at
     0xEFC0. Knock sum BELOW low threshold -> eec6++ (clamped 0-255).
     Knock sum AT/ABOVE high threshold -> eec6-- (floor 0). i.e. eec6
     rises when NOT knocking and falls when knocking - it tracks
     "octane margin", counting up toward 255 during clean running and
     down toward 0 under sustained knock.
     IMPORTANT: this write uses COMPUTED pointer arithmetic
     ((ushort*)(bank<<0x10 | 0xeec6)), which is why it was INVISIBLE to
     the original static xref-to-0xeec6 search - only found via a
     bank-prefixed query ("1eec6" instead of plain "eec6"). Same blind
     spot class as knock_process_main's own callers and the RPM-writer
     search - worth remembering as a general technique for future
     "no writer found" dead ends in this ROM.
  3. octane_level_compute_from_knock_counter (was FUN_000252d3, ROM
     0x252d3), later in the same chain: reads eec6, force-zeroes it if
     f1fe bit3 OR f200 bit0 is set (meaning not yet decoded), passes the
     result through a shared clamp/scale helper (o2_downstream_clamp_by_
     type - also used elsewhere for O2 values, exact transform not
     re-derived), and writes the result to 0xEEC8/EEC9 = Octane Level.
     THIS IS THE CONFIRMED WRITER that was missing before. Also uses
     computed addressing for the eec8 write.
  4. octane_level_init_or_force_from_gear_table (was FUN_00024e80, ROM
     0x24e80) is a SEPARATE writer of eec6+eec8 together (sets both to
     the same value in one pass) - NOT part of the dispatcher's own
     call chain, called from elsewhere (caller not yet identified).
     Driven by the f1fc gear-state index (confirmed session 3) via a
     4th per-gear ROM table at 0x2882 (index 1) when f1f2 bit8 is clear,
     or forces Octane Level to 0xff (100%) when f0fe bit0 is set. This
     means Octane Level also has a gear-dependent init/override path
     independent of the knock hysteresis counter - worth investigating
     further if gear-specific octane/timing behavior matters.

[CONFIRMED] Obtained a complete decompile of knock_process_main (0x2933b) via
live Ghidra connection and traced every write in the function. Exact writes,
in order of appearance:

  RAM 0xF3A8 (word) / low byte 0xF3A9 = MUT knock_adc (ReqID 0x6A, "Evo7/8").
    Value: adc_read_channel_6 result (0xF15C) run through mul_fixedpoint_8
    across 3 piecewise ranges (x<0x5D, 0x5D<=x<0x65, x>=0x65), clamped to
    0xFF. This is genuinely the direct per-cylinder ADC-derived knock
    intensity value. STATUS: CONFIRMED, semantically accurate.

  RAM 0xF962 (word) / low byte 0xF963 = MUT knock_var (ReqID 0x6C).
    Value: the FINAL per-cylinder knock decision/trigger magnitude (variable
    "uVar6" in the decompile) after the full threshold/hysteresis branch
    chain (0/0x10/0x20/0x100-style state selection, then persistence-counter
    and anti-chatter logic against ROM thresholds 0xb4e/0xb62/0x8e).
    STATUS: CONFIRMED address, but this is a DERIVED decision value, not raw
    ADC variance - "var" is a misleading but not incorrect name.

  RAM 0xF966 (word) / low byte 0xF967 = MUT knock_base (ReqID 0x6B).
    Value: a snapshot of 0xF3AA (the clamped per-cylinder knock-retard
    output) taken BEFORE this cycle's own 0xF3AA update later in the
    function - i.e. it holds the PREVIOUS cycle's retard output, not an
    independent "base" reading. STATUS: CONFIRMED address/write-site,
    semantics is "previous-cycle retard snapshot".

  RAM 0xF968 (word) / low byte 0xF969 = MUT knock_change (ReqID 0x6D).
    Value: delta between the current and previous 0xF3A8 knock-ADC reading
    (uVar8 = current - previous, saturated at 0), or forced to 0 when
    F3A4 bit10 is clear or F3A6 bit0 is set. STATUS: CONFIRMED, and the name
    "knock_change" is an accurate semantic match (it IS a delta/rate value).

  RAM 0xF3B6 (word) / low byte 0xF3B7 = MUT knock_dynamics (ReqID 0x6E).
    Value: a persistence/duration counter - increments each cycle knock
    stays below the low threshold (clamped by ROM 0xb8e/0xf8e), resets to 0
    when the high threshold is exceeded (also sets RAM 0xF200 bit0, a
    lockup-style flag also consumed by tcu_rx_main_scheduler). STATUS:
    CONFIRMED, "dynamics" is a reasonable name for a state-duration tracker.

[REFUTED] MUT knock_flag (ReqID 0x6F, RAM 0xEFC2/EFC3) is NOT written by
knock_process_main - it isn't referenced anywhere in the function at all
(confirmed by full decompile, not just an xref miss). Traced the actual
writers via get_xrefs_to on 0xEFC2:
  - FUN_000252f9 (0x252f9): writes 0xEFC2 from ROM constant 0xB24 when
    (RAM 0xF17E - 0x80) >= ROM threshold 0xB22. 0xF17E is unrelated to knock
    sum/ADC - this was already flagged as a separate timer/flag feature in
    session 4.
  - tcu_rx_main_scheduler (0x2b5bd, near ROM offset 0xb6ea): writes the SAME
    ROM constant 0xB24 to 0xEFC2 when a filtered delta of 0xF13C (current)
    vs a shifted history register at 0xF142 exceeds ROM threshold 0xB20.
  Both writers are throttle/crank-transient delta detectors, not knock
  detectors. tcu_rx_main_scheduler also READS 0xEFC2 back (near ROM 0xb08b)
  to select between ROM thresholds 0xB1C and 0xB1E - consistent with 0xEFC2
  being a shared "large throttle/crank movement" mode-select flag used by
  the TCU scheduler and an unrelated timer feature, not a knock condition.
  CONCLUSION: the stock EvoScan profile's "knock_flag" label for MUT 0x6F is
  a genuine mislabel on this ECU, not just unconfirmed. Recommend removing it
  from the "Knock Control" grouping in the logging profile or renaming/
  flagging it clearly as suspect.
