=== General notes on H8/500 ===

=== Addressing modes ===
Sz: 1=word
    0=byte
 r: 0~7 = use R0~R7 register
Register Direct
  Rn		1 0 1 0 Sz r r r 	EA = Rn
Register Indirect
  @Rn		1 1 0 1 Sz r r r 	EA = 0:15  Rn
                                     16:23 DP/TP/EP
Register Indirect with Displacement
  @(d:8,Rn)	1 1 1 0 Sz r r r	EA = 0:15  (Rn + sign-extended displacement)
                                     16:23 DP/TP/EP
  @(d:16,Rn)	1 1 1 1 Sz r r r	EA = 0:15  (Rn + displacement)
                                     16:23 DP/TP/EP
Register Indirect with Pre/Post-Decrement
  @-Rn		1 0 1 1 Sz r r r    EA = 0:15  (Rn - (1 or 2, based on Sz))
                                     16:23 DP/TP/EP (ignored in Minimum mode)
  @Rn+		1 1 0 0 Sz r r r    EA = 0:15  Rn
                                     16:23 DP/TP/EP (ignored in Minimum mode)
Immediate
  #xx:8		0 0 0 0 0 1 0 0 	1-byte immediate
  #xx:16	0 0 0 0 1 1 0 0		2-byte immediate
Absolute
  @aa:8		0 0 0 0 Sz 1 0 1 	EA = 0:7   EA extension data
                                             8:15  BR
									 16:23 0x00
  @aa:16	0 0 0 1 Sz 1 0 1	EA = 0:15  EA extension data
                                             16:23 DP
PC-Relative
  disp		No EA field. Specified in op-code.
      :8                        	EA = 0:15  (PC + sign-extended displacement)
	                                     16:23 CP
      :16                       	EA = 0:15  (PC + displacement)
	                                     16:23 CP

Where a choice between DP/TP/EP is available, the choice is automatically determined by the choice of general register Rn. For R0~R3, DP is used. For R4~R5, EP is used. For R6~R6 (SP,BP), TP is used.

=== H8/520 and H8/538F variant bugs (found during H8/539F development) ===

--- h8520.pspec ---

BUG: ADC_ADI_vector is mapped to ram:0x00D0, which is the same address as
SCI1_ERI_vector. One of these is wrong. Cross-check against the H8/520 hardware
manual vector table to determine the correct address for ADC_ADI.

BUG: <default_memory_blocks> is commented out entirely. No RAM block is created
automatically when loading an H8/520 ROM. The block definitions inside the comment
(ram at 0x20000, ROM mirror at 0x0000) need to be verified against the H8/520
hardware manual and re-enabled once confirmed correct.

NOTE: The <context_data> block sets CP, DP, TP, EP, BR, FPH all to zero as static
global defaults. This is adequate for single-page ROMs but will misidentify
cross-page far calls in any multi-page H8/520 application. The H8/539F setup script
approach (setting context registers per address range at import time) is more correct
for paged ROMs.

--- h8538f.pspec ---

BUG: <data_space space="ram" /> declaration is missing. The H8/520 pspec and the
H8/539F pspec both have this. Without it Ghidra may not correctly identify the data
address space for the H8/538F language.

BUG: The RAM block is defined as initialized="true" with length 0xF000 starting at
0x0000. This covers code space, not just RAM. The block should be uninitialised and
sized to actual on-chip RAM only. Verify against H8/538 hardware manual section 2.

MISSING: No peripheral register symbols at all. The H8/538 shares the same peripheral
set as the H8/539F (SCI, ADC, ports, timers, interrupt controller, DTC, WDT,
multiplier) at the same I/O addresses. The full register map from h8539f.pspec can be
ported to h8538f.pspec directly, with the exception of the flash memory control
registers (FLMCR, FLM_EBR1, FLM_EBR2, FLMER, FLMSR) which are 539F-specific --
the 538 uses EPROM, not flash.

--- patternconstraints.xml ---

BUG: patterns.xml is only wired up to H8:BE:16:H8520Max. The H8:BE:32:H8539F
language is not listed, so the function-start byte patterns have no effect when
analysing 539F ROMs. Add an H8:BE:32:H8539F entry if the prologue pattern in
patterns.xml matches real ECU ROM function starts (verify first).
